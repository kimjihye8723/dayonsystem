## 1. 시스템 개요 (Overview)

### 핵심 설계 방향

#### 웹 반응형 구조 (Responsive Web)
- 데스크톱, 태블릿, 모바일 웹 환경에서 모두 자연스럽게 표시되어야 합니다.
- 모바일 웹에서도 기능 축소 없이 핵심 정보와 주요 기능을 확인할 수 있어야 합니다.
- 레이아웃은 해상도 변화에 따라 유연하게 재배치되도록 설계합니다.

#### 메뉴 단위 독립 동작 구조
- 각 메뉴(페이지)는 서로 의존하지 않고 독립적으로 동작할 수 있어야 합니다.
- 특정 메뉴 수정 또는 장애가 다른 메뉴의 동작에 영향을 주지 않도록 분리 설계를 원칙으로 합니다.
- 향후 신규 메뉴 추가 시 기존 메뉴에 최소한의 영향만 주도록 구조화합니다.

#### 스타일 표준화 및 CSS 클래스화
- 전체 UI 스타일은 공통 CSS 클래스 기준으로 관리합니다.
- 인라인 스타일, 임시 스타일, 페이지별 중복 스타일 사용을 지양합니다.
- 색상, 폰트, 여백, 버튼, 카드, 테이블 등은 일관된 디자인 시스템으로 유지합니다.

#### 객체화된 개발 구조
- 개발 소스는 특정 페이지에 종속되지 않도록 객체화·모듈화하여 구성합니다.
- 공통 로직, UI 컴포넌트, 유틸 함수는 재사용 가능하도록 분리합니다.
- “한 곳에서만 수정하면 전체에 반영되는 구조”를 목표로 합니다.

#### 무분별한 직접 수정 방지
- 다른 개발자가 임의로 소스를 복사·붙여넣기 하거나 페이지별로 즉흥적인 코딩을 하지 못하도록 구조적으로 제한합니다.
- 표준 컴포넌트와 공통 모듈을 사용하지 않는 방식의 개발은 지양합니다.
- 테스트는 사용자가 실행합니다. 기능 구현하면서 테스트하지 않고 최대한 시간 소모를 줄입니다.

#### 확장성과 호환성 중심 설계
- 향후 종목 추가, 분석 기능 확장, UI 개편이 가능하도록 유연한 구조를 유지합니다.
- 특정 환경이나 특정 화면에 종속되지 않는 호환 가능한 설계를 기본 원칙으로 합니다.

요약하면, 본 시스템은 **“모바일까지 고려한 반응형 웹 플랫폼”**이며, 유지보수·확장·협업에 강한 구조를 최우선으로 설계된 홈페이지입니다.

※ 매매 로직·신호 알고리즘의 상세 규칙은 본 문서에서 다루지 않으며, 별도 내부 기준 문서를 따릅니다.  
항상 최신 버전의 지침을 따르세요.

## 4. 설치 및 세팅 (Installation)


#### 4.2.1 백엔드 환경변수(.env) 표준
- 백엔드 실행 포트 및 DB 접속 정보는 **환경변수(.env)** 로 관리합니다.
- **보안 정책(11번)**: `.env` 및 비밀번호/토큰이 포함된 파일은 Git에 절대 커밋하지 않습니다.

예시:
```env
DB_HOST=114.108.180.228
DB_USER=blueeye
DB_PASS=blueeye0037!
DB_NAME=joot_cms
PORT=5000
```

- `PORT`: 백엔드 실행 포트 (기본값: `5000`)
- `DB_PASS` 키 이름은 프로젝트 코드에서 동일하게 사용해야 합니다.  
  (코드가 `DB_PASSWORD`를 기대한다면 **코드 또는 env 키를 일치**시켜야 합니다.)

#### 4.3.1 프론트 환경변수(.env) 표준 (Vite)
- Vite 기반 프론트에서는 환경변수 키가 **`VITE_`로 시작**해야 클라이언트 번들에 주입됩니다.
- 아래 값은 클라이언트에 포함될 수 있으므로, 외부 노출이 가능한 민감정보(계정/비번)는 원칙적으로 넣지 않는 것을 권장합니다.  
  (현재 값은 운영 편의상 포함되어 있으므로 **접근 통제/도메인 제한**을 반드시 병행합니다.)

예시:
```env
# WebRTC / RTSP Settings
VITE_RTSP_URL=rtsp://admin:blueeye0037!@smartgate2.cns-link.net:8554/profile2/media.smp
VITE_WEBRTC_GATEWAY_URL=https://cctv.mysmartgate.kr
VITE_WEBRTC_CHANNEL=2
```

## 7. 개발 및 운영 원칙 (Development Guidelines)

## 9. 배포 절차 (Deployment)

목표: **원본 소스 보호 + 변경 범위 최소 + 배포 후 즉시 복구 가능**

### 9.3 배포 방식 표준

실제 서버 환경이 Docker 기반인지(Compose) 직접 실행인지에 따라 아래 중 하나를 채택합니다.  
프로젝트 운영 표준은 **한 가지로 고정**하는 것을 권장합니다.

#### 9.3.1 (A안) Docker 기반 배포 표준
1) 배포 브랜치/코드 확보
- 릴리스 브랜치: `v4.3.0-release`
- 최신 코드 반영:
  - `git fetch`
  - `git checkout v4.3.0-release`
  - `git pull`

2) (선택) 프론트 빌드 산출물 생성
- 필요 시:
  - `npm ci`
  - `npm run build`

3) Docker 이미지 빌드 및 배포
- 필요 시: `docker compose build --no-cache`
- 배포: `docker compose up -d`

4) (선택) Nginx 갱신(사용 시)
- `nginx -t`
- `systemctl reload nginx`

5) 헬스 체크(필수)
- 백엔드 포트 응답 확인(문서 기준 9100)
- 프론트 라우팅/로그인 확인
- 핵심 메뉴 2~3개 스모크 테스트(“메뉴 독립 동작” 검증)

6) 장애 시 롤백(필수)
- 직전 정상 버전 태그/커밋으로 이동 후 재배포:
  - `git checkout <직전태그 또는 커밋>`
  - `docker compose up -d --build`
- 주의: “과거 소스를 가져와 현재 원본에 덮어쓰기” 방식 금지(8.1 취지)

#### 9.3.2 (B안) 직접 실행(서비스/pm2) 기반 배포 표준
1) 코드 업데이트
- `git fetch`
- `git checkout v4.3.0-release`
- `git pull`

2) 백엔드 의존성/빌드
- Node:
  - `npm ci`
  - 필요 시 `npm run build`
- .NET:
  - `dotnet restore`
  - `dotnet publish -c Release -o ./publish`
- Python:
  - 가상환경은 Git 제외(11번 정책)
  - `pip install -r requirements.txt`

3) 서비스 재시작
- systemd:
  - `systemctl restart <service>`
  - `systemctl status <service>`
- pm2:
  - `pm2 restart <name>`
  - `pm2 logs --lines 200`

4) 헬스 체크/스모크 테스트
- 포트 응답, 로그인, 주요 메뉴 확인

5) 롤백
- `git checkout <직전태그 또는 커밋>`
- 빌드/재시작 반복

### 9.4 배포 완료 후 필수 정리
